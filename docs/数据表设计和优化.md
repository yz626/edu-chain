# EduChain 数据表优化设计方案

## 一、优化原则

1. **不使用外键关联** - 通过应用层保证数据一致性
2. **数据表尽量精简** - 避免冗余字段，减少数据存储
3. **字段必要性** - 只保留必要字段，非必要信息通过JSON扩展字段存储

---

## 二、当前数据表问题分析

### 2.1 当前表统计

| 模块 | 表数量 | 主要问题 |
|------|--------|----------|
| 用户与认证 | 9 | 字段冗余，部分功能可合并 |
| 组织管理 | 3 | 基本合理 |
| 证书管理 | 5 | 证书主表字段过多(70+字段) |
| 区块链 | 4 | 基本合理 |
| 验证服务 | 3 | 部分表可合并 |
| 审计日志 | 3 | 日志表可统一 |
| 系统管理 | 5 | 部分功能可精简 |
| 统计 | 1 | 可移除 |
| **总计** | **33** | |

### 2.2 主要问题

1. **certificates表字段过多(70+字段)**
   - 包含过多冗余字段如signed_by_name、issued_by_name等
   - 区块链信息、上链信息与证书基本信息混在一起
   
2. **日志表冗余**
   - login_logs、operation_logs、api_access_logs功能有重叠
   - 可统一为审计日志表

3. **可移除的表**
   - `captchas` - 验证码可使用Redis
   - `daily_statistics` - 统计可实时计算或使用分析型数据库

4. **冗余字段**
   - 多处存储可关联获取的用户名/组织名

---

## 三、优化后的数据表设计

### 3.1 模块划分

| 序号 | 模块 | 表数量 | 核心表 |
|------|------|--------|--------|
| 1 | 用户与权限 | 6 | users, roles, permissions, user_roles |
| 2 | 组织架构 | 3 | organizations, organization_users, departments |
| 3 | 证书管理 | 4 | certificates, certificate_types, certificate_templates, certificate_batches |
| 4 | 区块链 | 3 | blockchain_networks, blockchain_transactions, smart_contracts |
| 5 | 验证服务 | 1 | verifications |
| 6 | 审计日志 | 1 | audit_logs |
| 7 | 系统管理 | 4 | system_configs, dictionaries, file_records, job_queues |
| **总计** | | **22** | |

**优化效果**: 从33张表减少到22张表，减少33%

## 四、详细表结构设计

### 4.1 用户与权限模块 (6张表)

#### 4.1.1 用户基础表 (users)

```sql
CREATE TABLE `users` (
    `id` VARCHAR(36) NOT NULL COMMENT '用户ID (UUID)',
    `username` VARCHAR(64) NOT NULL COMMENT '用户名',
    `email` VARCHAR(128) NOT NULL COMMENT '邮箱地址',
    `phone` VARCHAR(20) DEFAULT NULL COMMENT '手机号',
    `password_hash` VARCHAR(255) NOT NULL COMMENT '加密后的密码哈希',
    `salt` VARCHAR(64) DEFAULT NULL COMMENT '密码盐值',
    `status` TINYINT NOT NULL DEFAULT 1 COMMENT '用户状态: 1-正常, 2-禁用, 3-待审核, 4-锁定',
    `user_type` TINYINT NOT NULL DEFAULT 1 COMMENT '用户类型: 1-普通用户, 2-管理员, 3-系统用户',
    `source` TINYINT DEFAULT 1 COMMENT '注册来源: 1-注册, 2-导入, 3-第三方, 4-API',
    `last_login_at` DATETIME(3) DEFAULT NULL COMMENT '最后登录时间',
    `last_login_ip` VARCHAR(45) DEFAULT NULL COMMENT '最后登录IP',
    `login_count` INT DEFAULT 0 COMMENT '登录次数',
    `failed_attempts` TINYINT DEFAULT 0 COMMENT '失败尝试次数',
    `locked_until` DATETIME(3) DEFAULT NULL COMMENT '锁定截止时间',
    `email_verified` TINYINT(1) DEFAULT 0 COMMENT '邮箱是否已验证',
    `phone_verified` TINYINT(1) DEFAULT 0 COMMENT '手机是否已验证',
    `two_factor_enabled` TINYINT(1) DEFAULT 0 COMMENT '是否启用双因素认证',
    `two_factor_secret` VARCHAR(128) DEFAULT NULL COMMENT '双因素认证密钥',
    `created_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '创建时间',
    `updated_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '更新时间',
    `deleted_at` DATETIME(3) DEFAULT NULL COMMENT '删除时间',
    `version` INT DEFAULT 1 COMMENT '版本号 (乐观锁)',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_users_username` (`username`),
    UNIQUE KEY `uk_users_email` (`email`),
    UNIQUE KEY `uk_users_phone` (`phone`),
    KEY `idx_users_status` (`status`),
    KEY `idx_users_user_type` (`user_type`),
    KEY `idx_users_created_at` (`created_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户基础表';
```

**优化点**: 
- 移除冗余的用户扩展信息表(user_profiles)，将扩展字段合并到users表中
- 精简后保留核心用户信息，非必要信息存入extra_data JSON字段

## 六、优化总结

### 6.1 表数量变化

| 优化前 | 优化后 | 减少 |
|--------|--------|------|
| 33张 | 22张 | 11张 (33%) |

### 5.2 主要优化措施

1. **合并日志表**: login_logs、operation_logs、api_access_logs → audit_logs
2. **移除冗余表**: 
   - captchas → 使用Redis
   - daily_statistics → 实时计算
   - verification_requests → 合并到verifications
   - verification_reports → 文件存储
   - blockchain_certificates → 合并到blockchain_transactions
   - user_profiles → 合并到users
   - certificate_issuances → 合并到certificates
   - notifications → 使用消息队列
3. **精简主表字段**: certificates表字段从70+减少到55
4. **移除外键**: 保持无外键设计，通过应用层保证一致性

### 5.3 设计原则遵循

- ✅ 不使用外键关联
- ✅ 数据表尽量精简(必要字段)
- ✅ 避免数据冗余(通过关联查询获取非必要信息)
- ✅ 使用JSON字段存储扩展数据
